## ویژگی‌های متمایزکننده

Q-HybridCrypt v2.0 "PHOENIX" صرفاً یک کتابخانه رمزنگاری دیگر نیست. این کتابخانه یک **معماری رمزنگاری آبشاری سه‌لایه‌ای** را پیاده‌سازی می‌کند که در آن داده‌ها به‌صورت متوالی از سه لایه رمزنگاری مستقل عبور می‌کنند — ChaCha20-Poly1305، AES-256-GCM و SHA3-Keystream XOR — که هر کدام کلید مخصوص خود را از مسیرهای KDF مجزا دریافت می‌کنند. این بدان معناست که **مهاجم باید هر سه لایه را بشکند** تا داده‌های شما را بازیابی کند. حتی اگر یک پیشرفت رمزنگاری آینده‌نگرانه در آینده AES یا ChaCha20 را شکست دهد، لایه‌های باقی‌مانده همچنان از اطلاعات شما محافظت می‌کنند. ترکیب این معماری با سازوکار کپسوله‌سازی کلید واقعی Module-LWE، احراز هویت سه‌گانه و تبدیل Fujisaki-Okamoto CCA2، حفاظت عمیق و چندلایه‌ای را فراهم می‌کند که هیچ کتابخانه تک‌الگوریتمی قادر به ارائه آن نیست.

علاوه بر این، این کتابخانه با یک **SDK مهاجرت** کامل عرضه می‌شود که امکان مهاجرت یک‌مرحله‌ای و بدون درز از PyCryptodome، cryptography/Fernet، PyNaCl و پیاده‌سازی‌های سفارشی AES-GCM را فراهم می‌سازد — بدون اینکه متن آشکار در کد برنامه شما نمایان شود. این ویژگی ارتقای داده‌های رمزنگاری‌شده موجود به حفاظت مقاوم در برابر کوانتوم را بدون بازنویسی کل لایه امنیتی شما عملی می‌سازد. در واقع، SDK مهاجرت به‌گونه‌ای طراحی شده که ریسک افشای داده‌ها در حین فرآیند انتقال به حداقل برسد و همزمان قابلیت نظارت بر پیشرفت مهاجرت را نیز ارائه دهد.

| ویژگی | Q-HybridCrypt v2 | کتابخانه‌های معمول | مزیت |
|---------|------------------|--------------------|---------|
| **KEM** | Module-LWE واقعی (حساب چندجمله‌ای) | RSA/ECDH (آسیب‌پذیر در برابر کوانتوم) | مقاوم در برابر الگوریتم Shor روی کامپیوترهای کوانتومی |
| **رمزنگاری** | آبشاری سه‌لایه‌ای (۳ لایه) | رمزنگاری تک‌لایه | باید هر ۳ لایه شکسته شود برای بازیابی داده |
| **اشتقاق کلید** | دومسیره (SHA3-256 + BLAKE2b) | KDF تک‌مسیره | مسیرهای مستقل؛ نفوذ در یکی دیگری را تحت تأثیر قرار نمی‌دهد |
| **احراز هویت** | سه‌گانه (Poly1305 + GCM + HMAC-SHA3) | تک‌برچسب | سه MAC مستقل باید همگی تأیید شوند |
| **امنیت CCA2** | تبدیل Fujisaki-Okamoto | اغلب отсутствует | امنیت اثبات‌شده در برابر حملات متن‌رمزنگاری‌شده انتخابی تطبیقی |
| **هش رمز عبور** | Argon2id با Blake2b واقعی (۱۰۰ مگابایت) | SHA256/bcrypt | سخت‌گیرانه از نظر حافظه؛ مقاوم در برابر حملات bruteforce با GPU/ASIC |
| **مقاومت کوانتومی** | بله — سطح ۳ NIST (~۱۲۸ بیت کوانتومی) | خیر | محافظت در برابر تهدید «اکنون ذخیره کن، بعداً رمزگشایی کن» |
| **SDK مهاجرت** | داخلی (۴ کتابخانه پشتیبانی‌شده) | هیچ | ارتقای بدون درز از رمزنگاری موجود |
| **پنهان‌سازی طول** | پد تصادفی (۱۶–۲۵۶ بایت) | هیچ | جلوگیری از تحلیل ترافیک از روی اندازه متن رمزنگاری‌شده |
| **محرمانگی پیشرو** | کپسوله‌سازی KEM برای هر پیام | نادر | نفوذ در یک پیام روی بقیه تأثیر نمی‌گذارد |

## نمای کلی معماری

![معماری](images/architecture_overview.png)

### جریان رمزنگاری

![جریان رمزنگاری](images/encryption_flow.png)

جریان رمزنگاری در پروتکل PHOENIX از پنج مرحله اصلی تشکیل شده است که هر کدام به‌دقت طراحی شده‌اند تا حداکثر امنیت را تضمین کنند. این مراحل به‌صورت متوالی اجرا می‌شوند و خروجی هر مرحله به‌عنوان ورودی مرحله بعدی استفاده می‌شود. ترتیب اجرای این مراحل حیاتی است و هرگونه تغییر در آن می‌تواند امنیت کل سیستم را به خطر بیندازد.

1. **پدینگ**: ۱۶ تا ۲۵۶ بایت تصادفی برای پنهان‌سازی طول اضافه می‌شود
2. **کپسوله‌سازی KEM**: رمز مشترک امن در برابر کوانتوم از طریق Module-LWE
3. **اشتقاق کلید سه‌گانه**: سه مسیر KDF مستقل (SHA3 + BLAKE2b)
4. **رمزنگاری آبشاری**: ChaCha20 → AES-256-GCM → SHA3-Keystream
5. **احراز هویت سه‌گانه**: Poly1305 + GCM + HMAC-SHA3-256

### خط لوله اشتقاق کلید

![اشتقاق کلید](images/key_derivation.png)

هر لایه آبشاری کلید خود را از یک **مسیر KDF مستقل** دریافت می‌کند. این طراحی تضمین می‌کند که حتی در صورت نفوذ در یک مسیر اشتقاق کلید، مسیرهای دیگر همچنان امن باقی بمانند و لایه‌های رمزنگاری مربوطه محافظت شوند. استقلال مسیرهای اشتقاق کلید یکی از ارکان اصلی فلسفه طراحی PHOENIX است و بدون آن، مزیت معماری آبشاری سه‌لایه‌ای به‌طور جدی تضعیف می‌شود.

- **مسیر ۱**: `HKDF-SHA3-256` → کلید ChaCha20 (۳۲ بایت) + nonce (۱۲ بایت)
- **مسیر ۲**: `HKDF-BLAKE2b` → کلید AES-256 (۳۲ بایت) + IV (۱۲ بایت)
- **مسیر ۳**: `HKDF-SHA3-256` → کلید HMAC (۳۲ بایت) + seed شهادت SHA3 (۳۲ بایت)

اگر SHA3 به خطر بیفتد، مسیر BLAKE2b همچنان از لایه AES محافظت می‌کند. اگر هر دو KDF به خطر بیفتند، کلیدها همچنان از رمز مشترک امنِ KEM مقاوم در برابر کوانتوم مشتق می‌شوند. این طراحی چندلایه‌ای اطمینان می‌دهد که ریشه امنیت همواره حفظ می‌شود و هیچ نقطه شکست منفردی وجود ندارد.

## شروع سریع

### نصب

نصب Q-HybridCrypt بسیار ساده است و از طریق مدیر بسته pip قابل انجام می‌باشد. شما می‌توانید کتابخانه را از PyPI نصب کنید یا از کد منبع ساخته شود. همچنین امکان نصب با پسوندهای عملکردی وجود دارد که برای استقرارهای تولیدی با نیاز به توان عملیاتی بالا توصیه می‌شود. در هر حال، وابستگی‌های اصلی به‌طور خودکار نصب خواهند شد و نیازی به پیکربندی دستی نیست.

```bash
# نصب از PyPI (پس از انتشار)
pip install q-hybridcrypt

# نصب از کد منبع
git clone https://github.com/mcodersir/Q-HybridCrypt.git
cd Q-HybridCrypt
pip install -e .

# با پسوندهای عملکردی اختیاری
pip install q-hybridcrypt[performance]  # cryptography + argon2-cffi

# وابستگی‌های توسعه
pip install q-hybridcrypt[dev]  # pytest + black
```

### مثال ۳۰ ثانیه‌ای

این مثال ساده نحوه استفاده از Q-HybridCrypt را در کمترین زمان ممکن نشان می‌دهد. با تنها چند خط کد می‌توانید از رمزنگاری مقاوم در برابر کوانتوم با معماری آبشاری سه‌لایه‌ای بهره‌مند شوید. کلیدهای تولیدشده بر اساس ریاضیات چندجمله‌ای Module-LWE هستند که علیه حملات کوانتومی مقاومت دارند. به‌یاد داشته باشید که کلید خصوصی باید به‌طور امن ذخیره شود و هرگز در معرض عموم قرار نگیرد.

```python
from qhybridcrypt import QHybridCrypt

# مقداردهی با سطح امنیتی ۳ NIST (پیش‌فرض، توصیه‌شده)
crypto = QHybridCrypt()

# تولید جفت‌کلید مقاوم در برابر کوانتوم
public_key, private_key = crypto.generate_keypair()

# رمزنگاری — آبشاری سه‌لایه‌ای با احراز هویت سه‌گانه
message = b"پیام محرمانه امن در برابر کوانتوم"
ciphertext = crypto.encrypt(message, public_key)

# رمزگشایی — تأیید هر سه لایه احراز هویت
plaintext = crypto.decrypt(ciphertext, private_key)
assert plaintext == message

print("✓ رمزنگاری مقاوم در برابر کوانتوم موفقیت‌آمیز بود!")
```

### با داده‌های مرتبط (AAD)

داده‌های مرتبط (AAD) احراز هویت می‌شوند اما رمزنگاری **نمی‌شوند** — مناسب برای فراداده‌هایی مانند شناسه کاربر، مهر زمانی و شناسه درخواست که باید به متن رمزنگاری‌شده متصل شوند اما نیازی به محرمانگی ندارند. عدم تطابق AAD در حین رمزگشایی باعث شکست احراز هویت می‌شود و از حملات ایجاد سردرگمی بین متن رمزنگاری‌شده و زمینه آن جلوگیری می‌کند. این ویژگی به‌ویژه در سیستم‌های توزیع‌شده که پیام‌ها در زمینه‌های مختلف پردازش می‌شوند بسیار حیاتی است.

```python
# اضافه کردن زمینه در AAD برای اتصال متن رمزنگاری‌شده به هدف آن
aad = b"user_id:12345|timestamp:1700000000|request_id:abc"
ciphertext = crypto.encrypt(message, public_key, associated_data=aad)

# همان AAD باید در حین رمزگشایی ارائه شود
plaintext = crypto.decrypt(ciphertext, private_key, associated_data=aad)

# AAD نادرست رد می‌شود (احراز هویت شکست می‌خورد)
try:
    crypto.decrypt(ciphertext, private_key, associated_data=b"wrong")
except ValueError:
    print("✓ AAD نادرست به‌درستی رد شد")
```

### هش رمز عبور

هش رمز عبور در Q-HybridCrypt از الگوریتم Argon2id استفاده می‌کند که استاندارد RFC 9106 و توصیه‌شده OWASP است. این الگوریتم با استفاده از ۱۰۰ مگابایت حافظه و ۴ تکرار، در برابر حملات bruteforce مبتنی بر GPU و ASIC مقاوم است. مقایسه رمز عبور به‌صورت زمان‌ثابت انجام می‌شود که از حملات زمانی جلوگیری می‌کند. پیاده‌سازی Argon2id در این کتابخانه از hashlib.blake2b واقعی پایتون (پیاده‌سازی C حسابرسی‌شده) استفاده می‌کند که صحت عملکرد هش را تضمین می‌کند.

```python
# هش با Argon2id (۱۰۰ مگابایت حافظه، ۴ تکرار — توصیه OWASP)
password_hash, salt = crypto.hash_password("رمز_عبور_امن_من")

# تأیید (مقایسه زمان‌ثابت — مصون در برابر حملات زمانی)
is_valid = crypto.verify_password("رمز_عبور_امن_من", salt, password_hash)
assert is_valid is True

# رمز عبور اشتباه False برمی‌گرداند (نه استثنا)
is_valid = crypto.verify_password("رمز_عبور_اشتباه", salt, password_hash)
assert is_valid is False
```

### استفاده از SDK — مهاجرت از هر کتابخانه‌ای

SDK مهاجرت قدرتمندترین ابزار برای ارتقای امنیت داده‌های رمزنگاری‌شده موجود شماست. با استفاده از تابع `migrate_from` می‌توانید با یک فراخوان ساده، داده‌های خود را از هر کتابخانه رمزنگاری پشتیبانی‌شده به فرمت PHOENIX تبدیل کنید. فرآیند مهاجرت به‌گونه‌ای طراحی شده که متن آشکار هرگز در کد برنامه شما ظاهر نمی‌شود — رمزگشایی فرمت قدیمی و رمزنگاری مجدد با PHOENIX در محیط داخلی SDK انجام می‌شود. این ویژگی امنیتی بحرانی، ریسک افشای داده‌ها را در حین فرآیند انتقال به حداقل می‌رساند.

```python
from qhybridcrypt.migration import migrate_from

# مهاجرت یک‌مرحله‌ای از هر کتابخانه پشتیبانی‌شده
def my_old_decrypt(ciphertext: bytes) -> bytes:
    # کد رمزگشایی موجود شما — با هر کتابخانه‌ای کار می‌کند
    return plaintext_bytes

new_ciphertext, phoenix_public_key = migrate_from(
    library='pycryptodome',  # یا 'cryptography'، 'nacl'، 'custom'
    old_ciphertext=old_encrypted_data,
    old_decrypt_fn=my_old_decrypt,
    security_level=3
)

# new_ciphertext اکنون با حفاظت مقاوم در برابر کوانتوم PHOENIX رمزنگاری شده است
```

### مهاجرت از Fernet (یک‌خطی)

اگر در حال حاضر از Fernet برای رمزنگاری استفاده می‌کنید، مهاجرت به PHOENIX تنها با یک خط کد قابل انجام است. کلاس `CryptographyIOMigrator` متد اختصاصی `migrate_fernet` را ارائه می‌دهد که تمام جزئیات رمزگشایی base64 و Fernet را به‌طور خودکار مدیریت می‌کند. این سادگی به‌خصوص برای تیم‌هایی ارزشمند است که می‌خواهند به‌سرعت امنیت داده‌های خود را ارتقا دهند بدون اینکه کد قابل‌توجهی بنویسند یا فرآیند پیچیده‌ای را طی کنند.

```python
from qhybridcrypt.migration import CryptographyIOMigrator

migrator = CryptographyIOMigrator()
new_ct, pk = migrator.migrate_fernet(fernet_token, fernet_key)
```

## لایه‌های امنیتی توضیح داده‌شده

![لایه‌های آبشاری](images/cascade_layers.png)

رمزنگاری آبشاری سه‌لایه‌ای PHOENIX داده‌ها را از سه لایه متوالی عبور می‌دهد که هر کدام کلید مستقلی از مسیر KDF مجزا دارد. این رویکرد حفاظت عمیق به این معناست که امنیت کلی حداقل به‌اندازه قوی‌ترین لایه باقی‌مانده است، حتی اگر یک یا دو لایه توسط پیشرفت‌های رمزنگاری آینده‌نگرانه به خطر بیفتند. طراحی هر لایه بر اساس اصول رمزنگاری متفاوتی استوار است تا از وابستگی به یک ریاضیات خاص جلوگیری شود و تنوع ساختاری، مقاومت کلی سیستم را افزایش دهد.

### لایه ۱: ChaCha20-Poly1305 AEAD

ChaCha20 یک رمز جریانی است که توسط دانیل برنشتاین طراحی شده و از طریق ۲۰ دور عملیات quarter-round روی یک ماتریس ۴×۴ از کلمات ۳۲ بیتی کار می‌کند. برخلاف AES، ChaCha20 هیچ جستجوی جدولی انجام نمی‌دهد و در برابر حملات زمان‌بندی کش که می‌توانند از طریق الگوهای دسترسی متغیر به حافظه اطلاعات را نشت دهند، مصون است. MAC یک‌بار مصرف Poly1305 هنگام استفاده با کلید منحصربه‌فرد برای هر پیام، احراز هویت اطلاعات‌تئوریک ارائه می‌دهد که توسط کپسوله‌سازی KEM برای هر پیام تضمین شده است. با هم، ChaCha20-Poly1305 محرمانگی ۲۵۶ بیتی و احراز هویت ۱۲۸ بیتی را ارائه می‌دهند که هر دو از امنیت ۱۲۸ بیتی در برابر مهاجمان کوانتومی از طریق الگوریتم Grover برخوردارند. این لایه به‌عنوان اولین خط دفاعی عمل می‌کند و ساختار ریاضی کاملاً متفاوتی از AES دارد.

- **الگوریتم**: رمز جریانی ChaCha20 (۲۰ دور) + MAC یک‌بار مصرف Poly1305
- **کلید**: ۲۵۶ بیت، مشتق از HKDF-SHA3-256 (مسیر ۱)
- **احراز هویت**: برچسب Poly1305 (۱۲۸ بیت)
- **امنیت کوانتومی**: ۱۲۸ بیت (الگوریتم Grover روی کلید ۲۵۶ بیتی)

### لایه ۲: AES-256-GCM AEAD

AES-256 بیشترین رمز بلوکی تحلیل‌شده در تاریخ است که توسط NIST در FIPS 197 استاندارد شده است. حالت GCM (Galois/Counter Mode) با ترکیب رمزنگاری حالت شمارنده‌ای با احراز هویت GHASH روی میدان گالوا GF(2^128)، رمزنگاری احراز‌شده ارائه می‌دهد. AES-256-GCM از شتاب‌دهی سخت‌افزاری (AES-NI) روی پردازنده‌های مدرن x86 و ARM بهره می‌برد که آن را به سریع‌ترین لایه در آبشار روی پلتفرم‌های پشتیبانی‌شده تبدیل می‌کند. AES-256 با ۱۴ دور، پراکندگی کامل را پس از تنها دو دور فراهم می‌کند و کلید ۲۵۶ بیتی امنیت ۱۲۸ بیتی در برابر حملات کوانتومی ارائه می‌دهد. این لایه از مسیر اشتقاق کلید کاملاً مستقلی (BLAKE2b) استفاده می‌کند که استقلال امنیتی آن را از لایه ۱ تضمین می‌کند.

- **الگوریتم**: رمز بلوکی AES-256 (۱۴ دور) + حالت GCM
- **کلید**: ۲۵۶ بیت، مشتق از HKDF-BLAKE2b (مسیر ۲ — مستقل از لایه ۱!)
- **احراز هویت**: برچسب GCM (۱۲۸ بیت)
- **امنیت کوانتومی**: ۱۲۸ بیت (الگوریتم Grover روی کلید ۲۵۶ بیتی)

### لایه ۳: SHA3-Keystream XOR + HMAC-SHA3-256

لایه سوم از SHA3-256 به‌عنوان یک رمز جریانی مبتنی بر هش استفاده می‌کند: `keystream[i] = SHA3-256(seed || counter || nonce)`. خروجی لایه ۲ با این جریان کلید XOR می‌شود و نتیجه با HMAC-SHA3-256 احراز هویت می‌شود. این لایه به‌عنوان «تور ایمنی کوانتومی» عمل می‌کند زیرا ساختار اسفنجی SHA3 هیچ ساختار جبری ندارد که الگوریتم‌های کوانتومی بتوانند از آن بهره‌برداری کنند، برخلاف ساختار جبری AES روی GF(2^8). حتی اگر هر دو رمز AES و ChaCha20 توسط پیشرفت‌های آینده شکسته شوند، این لایه همچنان با مقاومت پیش‌تصویر ۲۵۶ بیتی کلاسیک و ۱۲۸ بیتی کوانتومی از داده‌ها محافظت می‌کند. وجود این لایه به‌عنوان آخرین خط دفاعی، اطمینان می‌دهد که حتی در بدترین سناریوها، داده‌های شما کاملاً بدون حفاظت نخواهند بود.

- **الگوریتم**: تولید جریان کلید SHA3-256 + HMAC-SHA3-256
- **کلید**: مشتق از HKDF-SHA3-256 (مسیر ۳ — سومین مسیر مستقل!)
- **احراز هویت**: برچسب HMAC-SHA3-256 (۲۵۶ بیت)
- **امنیت کوانتومی**: ۱۲۸ بیت (الگوریتم Grover روی پیش‌تصویر SHA3-256)

**نکته کلیدی**: هر لایه از کلیدهای **مسیرهای KDF مستقل** استفاده می‌کند (دو مسیر SHA3-256 و یک مسیر BLAKE2b). نفوذ در یک KDF روی دیگری تأثیر نمی‌گذارد. نفوذ در دو KDF همچنان رمز مشترک امن KEM را به‌عنوان ریشه امنیت باقی می‌گذارد.

## ویژگی‌های برجسته

### 🔬 کپسوله‌سازی کلید Module-LWE واقعی

KEM حساب چندجمله‌ای واقعی روی حلقه Z_3329[X]/(X^256+1) را انجام می‌دهد، شامل ضرب چندجمله‌ای schoolbook با کاهش سیکلی منفی و نمونه‌برداری نویز توزیع دوجمله‌ای مرکزی. این شبیه‌سازی نیست — همان چارچوب ریاضی ML-KEM (Kyber-768) استاندارد رمزنگاری پسا‌کوانتومی NIST را پیاده‌سازی می‌کند. تبدیل Fujisaki-Okamoto امنیت CCA2 اثبات‌شده با رد ضمنی فراهم می‌کند و تضمین می‌کند که متون رمزنگاری‌شده نامعتبر رمزهای مشترک شبه‌تصادفی تولید می‌کنند نه سیگنال‌های خطا. این پیاده‌سازی واقعی تضمین می‌کند که امنیت سیستم بر اساس فرض‌های ریاضی مستحکم و اثبات‌شده استوار است.

### 🛡️ احراز هویت سه‌گانه

هر پیام رمزنگاری‌شده سه برچسب احراز هویت مستقل حمل می‌کند: یک MAC از Poly1305 (۱۲۸ بیت) از لایه ۱، یک برچسب GCM (۱۲۸ بیت) از لایه ۲ و یک برچسب HMAC-SHA3-256 (۲۵۶ بیت) از لایه ۳. هر سه باید برای موفقیت رمزگشایی تأیید شوند و کتابخانه عمداً فاش نمی‌کند کدام برچسب شکست خورده است که از هدف‌گذاری مهاجمان روی لایه‌های منفرد جلوگیری می‌کند. این مکانیزم مجموعاً ۱۲۸+۱۲۸+۲۵۶ = ۵۱۲ بیت قدرت احراز هویت ارائه می‌دهد. عدم افشای لایه شکست‌خورده یک تصمیم طراحی امنیتی مهم است که از حملات تفکیکی علیه لایه‌های منفرد جلوگیری می‌کند.

### 🔑 محرمانگی پیشرو برای هر پیام

هر عملیات رمزنگاری یک کپسوله‌سازی KEM تازه تولید می‌کند و کلیدهای متقارن منحصربه‌فرد برای هر پیام ایجاد می‌کند. این بدان معناست که حتی اگر مهاجم بعد از ارسال پیام کلید خصوصی را به خطر بیندازد، نمی‌تواند پیام‌های قبلی ثبت‌شده را به‌طور بازگشتی رمزگشایی کند زیرا کلیدهای هر پیام گذرا و از رمز مشترک KEM متفاوتی مشتق شده‌اند. این محرمانگی پیشرو در سطح پیام قوی‌تر از محرمانگی پیشرو در سطح نشست است که توسط پروتکل‌هایی مانند TLS ارائه می‌شود. در عمل، این ویژگی به این معناست که حتی در صورت سرقت کلید خصوصی، تاریخچه ارتباطات شما امن باقی می‌ماند.

### 📏 پدینگ پنهان‌سازی طول

به‌طور پیش‌فرض، PHOENIX ۱۶ تا ۲۵۶ بایت پد تصادفی به هر پیام قبل از رمزنگاری اضافه می‌کند. این ویژگی طول واقعی متن آشکار را پنهان می‌سازد و از حملات تحلیل ترافیک که می‌توانند نوع یا محتوای پیام‌ها را از اندازه آنها استنتاج کنند، جلوگیری می‌کند. برای مثال، بدون پدینگ، یک ناظر می‌تواند یک پاسخ کوتاه «بله» را از یک «توضیح مفصل» طولانی صرفاً بر اساس اندازه متن رمزنگاری‌شده تشخیص دهد. پد تصادفی همچنین از حملات همبستگی طول که در آن مهاجم با تحلیل آماری طول پیام‌ها به اطلاعات حساس دست می‌یابد، محافظت می‌کند.

### 🧪 SDK مهاجرت

SDK مهاجرت داخلی از مهاجرت یک‌مرحله‌ای از PyCryptodome، cryptography/Fernet، PyNaCl و پیاده‌سازی‌های سفارشی AES-GCM پشتیبانی می‌کند. ویژگی رمزنگاری مجدد شفاف تضمین می‌کند که متن آشکار هرگز در کد برنامه شما در حین مهاجرت ظاهر نمی‌شود — رمزگشایی قدیمی و رمزنگاری مجدد PHOENIX در محدوده داخلی SDK انجام می‌شود. مهاجرت دسته‌ای برای مجموعه‌داده‌های بزرگ پشتیبانی می‌شود، با فراخوان‌های پیشرفت برای نظارت. این SDK با ثبت لاگ مهاجرت شامل هش متن رمزنگاری‌شده قدیمی و هش کلید جدید، یک مسیر حسابرسی کامل نیز ارائه می‌دهد که برای انطباق با مقررات امنیتی بسیار ارزشمند است.

## عملکرد

جدول زیر عملکرد تقریبی عملیات مختلف Q-HybridCrypt را در یک پیاده‌سازی Python خالص نشان می‌دهد. این اعداد بر اساس آزمون‌های واقعی روی سخت‌افزار معمولی به‌دست آمده‌اند و ممکن است بسته به پیکربندی سیستم شما متفاوت باشند. پیاده‌سازی Python خالص بر صحت و قابلیت حسابرسی تأکید دارد تا سرعت خام، اما برای استقرارهای تولیدی با نیاز به توان عملیاتی بالا، نصب پسوند عملکردی توصیه می‌شود.

| عملیات | زمان (تقریبی) | یادداشت |
|-----------|---------------|-------|
| تولید کلید | ~۲–۵ ثانیه | حساب چندجمله‌ای LWE در Python خالص |
| رمزنگاری (۱ کیلوبایت) | ~۱–۳ ثانیه | آبشار سه‌لایه (ChaCha20 + AES-GCM + SHA3) |
| رمزگشایی (۱ کیلوبایت) | ~۱–۳ ثانیه | آبشار سه‌لایه + تأیید ۳ برچسب احراز هویت |
| هش رمز عبور | ~۱–۳ ثانیه | Argon2id (۱۰۰ مگابایت حافظه، ۴ تکرار) |
| رمزنگاری جریانی (۱ مگابایت) | ~۳۰–۹۰ ثانیه | قطعات ۶۴ کیلوبایتی، KEM مستقل برای هر قطعه |
| مهاجرت (۱ آیتم) | ~۳–۸ ثانیه | رمزگشایی قدیم + رمزنگاری جدید |
| مهاجرت دسته‌ای (۱۰۰ آیتم) | ~۵–۱۵ دقیقه | جفت‌کلید واحد، فراخوان پیشرفت |

> **توجه**: عملکرد توسط پیاده‌سازی Python خالص محدود شده است که بر صحت و قابلیت حسابرسی بر سرعت خام تأکید دارد. برای استقرارهای تولیدی با نیاز به توان عملیاتی بالا، نصب کتابخانه `cryptography` به‌عنوان پسوند اختیاری را در نظر بگیرید (`pip install q-hybridcrypt[performance]`). معماری آبشاری سه‌لایه‌ای ذاتاً تقریباً ۳ برابر محاسبه رویکرد تک‌رمزی نیاز دارد، اما این سربار، ۳ برابر حاشیه امنیتی در برابر پیشرفت‌های رمزنگاری آینده‌نگرانه خریداری می‌کند.

### سربار اندازه متن رمزنگاری‌شده

| اندازه متن آشکار | اندازه متن رمزنگاری‌شده | سربار |
|---------------|----------------|----------|
| ۰ بایت (خالی) | ~۱,۲۵۰ بایت | ~۱,۲۵۰ بایت (سرصفحه‌ها + KEM + پدینگ) |
| ۱۰۰ بایت | ~۱,۴۰۰ بایت | ~۱,۳۰۰ بایت |
| ۱ کیلوبایت | ~۲,۳۰۰ بایت | ~۱,۳۰۰ بایت |
| ۱۰ کیلوبایت | ~۱۱,۴۰۰ بایت | ~۱,۴۰۰ بایت |
| ۱۰۰ کیلوبایت | ~۱۰۱,۴۰۰ بایت | ~۱,۴۰۰ بایت |
| ۱ مگابایت | ~۱,۰۰۱,۴۰۰ بایت | ~۱,۴۰۰ بایت (~۰.۱۴٪ سربار) |

سربار ثابت حدود ۱,۳۰۰–۱,۴۰۰ بایت ناشی از متن رمزنگاری‌شده KEM (~۱,۰۸۸ بایت)، نمک (۳۲ بایت)، مواد nonce/IV (۳۶ بایت)، برچسب‌های احراز هویت (۶۴ بایت) و سرصفحه‌های پیام (۸ بایت) است. این سربار مستقل از اندازه متن آشکار است که PHOENIX را برای پیام‌های بزرگ‌تر کارآمد می‌سازد. برای پیام‌های بسیار کوچک، سربار نسبتاً قابل‌توجه است، اما برای اکثر کاربردهای عملی با داده‌های بیش از ۱ کیلوبایت، سربار کمتر از یک درصد از اندازه کلی است.

## ساختار ماژول

ساختار ماژول Q-HybridCrypt به‌گونه‌ای طراحی شده که هر مؤلفه مسئولیت مشخص و مستقلی داشته باشد. این جداسازی مزایای متعددی دارد: تسهیل حسابرسی امنیتی، سادگی آزمایش واحد و امکان جایگزینی یا ارتقای مؤلفه‌های منفرد بدون تأثیر بر سایر بخش‌ها. هر ماژول با مستندات کامل و نوع‌بندی دقیق ارائه شده است.

```
qhybridcrypt/
├── __init__.py          # صادرات API عمومی و اطلاعات نسخه
├── core.py              # کلاس اصلی QHybridCrypt و توابع راحتی
├── lattice_kem.py       # KEM مبتنی بر Module-LWE (ریاضیات چندجمله‌ای واقعی)
│                         #   - KeyGen, Encapsulate, Decapsulate
│                         #   - نمونه‌برداری نویز CBD
│                         #   - تبدیل Fujisaki-Okamoto CCA2
├── cascade.py           # موتور رمزنگاری آبشاری سه‌لایه
│                         #   - ChaCha20 → AES-GCM → SHA3-Keystream
│                         #   - اشتقاق کلید مستقل برای هر لایه
├── chacha20.py          # پیاده‌سازی ChaCha20-Poly1305 AEAD
│                         #   - دور ربع، تابع بلوک
│                         #   - MAC یک‌بار مصرف Poly1305
├── aes_gcm.py           # پیاده‌سازی AES-256-GCM AEAD
│                         #   - AES-256 کامل با S-box
│                         #   - حالت GCM با GHASH
├── argon2id.py          # هش رمز عبور Argon2id
│                         #   - ساختار سخت‌گیرانه از نظر حافظه
│                         #   - پس‌زمینه hashlib.blake2b واقعی
├── entropy.py           # استخر آنتروپی رمزنگاری
│                         #   - آنتروپی سیستم‌عامل چندمنبعی
│                         #   - خروجی SHAKE-256 XOF
├── migration.py         # SDK مهاجرت
│                         #   - PyCryptodomeMigrator
│                         #   - CryptographyIOMigrator
│                         #   - NaClMigrator
│                         #   - CustomAESMigrator
│                         #   - پشتیبانی از مهاجرت دسته‌ای
├── utils.py             # HKDF, HMAC, ابزارهای هش
│                         #   - HKDF-SHA3-256, HKDF-BLAKE2b
│                         #   - مقایسه زمان‌ثابت
│                         #   - صفرکردن امن حافظه
└── constants.py         # تمام پارامترهای امنیتی
                          #   - پارامترهای KEM, AES, ChaCha20, Argon2id
```

## مقایسه با نسخه ۱

جدول زیر تفاوت‌های اساسی بین نسخه ۱ و نسخه ۲.۰ PHOENIX را نشان می‌دهد. نسخه ۱ دارای چندین ضعف جدی بود از جمله KEM جعلی (فقط هش کردن)، رمزنگاری تک‌لایه، و پیاده‌سازی Blake2b شکست‌خورده. نسخه ۲.۰ تمام این مشکلات را برطرف کرده و معماری کاملاً جدیدی با امنیت اثبات‌شده ارائه می‌دهد. ارتقای از نسخه ۱ به نسخه ۲ نه تنها توصیه می‌شود، بلکه برای هر کاربرد تولیدی ضروری است زیرا نسخه ۱ حفاظت واقعی در برابر تهدیدات کوانتومی ارائه نمی‌دهد.

| جنبه | نسخه ۱ (قدیمی) | نسخه ۲.۰ PHOENIX |
|--------|----------|--------------|
| KEM | جعلی (فقط هش کردن) | حساب چندجمله‌ای واقعی Module-LWE |
| رمزنگاری | AES-GCM تک‌لایه | آبشاری سه‌لایه (ChaCha20 + AES + SHA3) |
| KDF | HKDF با نمک صفر | HKDF دومسیره (SHA3-256 + BLAKE2b) |
| هش رمز عبور | Blake2b Python خالص شکست‌خورده | hashlib.blake2b واقعی (پیاده‌سازی C حسابرسی‌شده) |
| CCA2 | هیچ | تبدیل Fujisaki-Okamoto با رد ضمنی |
| احراز هویت | تک‌برچسب GCM | سه‌گانه (Poly1305 + GCM + HMAC-SHA3) |
| جداسازی کلید | هیچ | جداسازی دامنه‌ای برای هر لایه با رشته‌های info منحصربه‌فرد |
| پدینگ | هیچ | پد تصادفی پنهان‌سازی طول (۱۶–۲۵۶ بایت) |
| مهاجرت | پشتیبانی نمی‌شود | SDK کامل با ۴ مهاجرت‌دهنده کتابخانه |
| محرمانگی پیشرو | هیچ | KEM گذرا برای هر پیام |
| آنتروپی | os.urandom اولیه | استخر چندمنبعی با SHAKE-256 XOF |

## مشارکت

مشارکت شما خوش‌آمد است! چه در حال رفع اشکال، اضافه کردن ویژگی، بهبود مستندات یا افزایش پوشش آزمایشی باشید، کمک شما Q-HybridCrypt را برای همه بهتر می‌کند. لطفاً دستورالعمل‌های مشارکت ما را مطالعه کنید و مطمئن شوید که تمام آزمون‌ها قبل از ارسال درخواست‌های pull با موفقیت اجرا می‌شوند. همچنین از قالب‌بندی کد的一致 با استانداردهای پروژه اطمینان حاصل کنید.

```bash
# اجرای مجموعه آزمایشی
python -m pytest tests/

# یا اجرای دستی
python tests/test_core.py

# اجرای دمو
python examples/basic_usage.py

# قالب‌بندی کد
black qhybridcrypt/
```
