## 概述

Q-HybridCrypt v2.0 "PHOENIX" 实现了一种**纵深防御**的密码学架构。与依赖单一加密算法（从而产生单点故障）的传统方案不同，PHOENIX 将三个独立的密码层链接在一起，每一层都拥有通过独立 KDF 路径派生的专属密钥。这种架构的核心思想是：即使某一个密码原语或密钥派生函数被完全攻破，其他层仍然能够独立保护数据安全，整体系统的安全性至少与最强的剩余层相当。

PHOENIX 架构的设计充分考虑了当前和未来的威胁形势。在经典计算威胁方面，三层级联提供了远超单层加密的安全余量；在量子计算威胁方面，Module-LWE KEM 基于被认为对量子计算机困难的问题，而所有对称密钥均为 256 位（通过 Grover 算法提供 128 位量子安全性）。Fujisaki-Okamoto 变换确保了 KEM 在自适应选择密文攻击下的安全性，三重认证机制则防止了密文被篡改或伪造的任何可能性。整个系统从密钥生成到密文输出的每一个环节都经过精心设计，确保不存在可被利用的安全薄弱点。

## 设计哲学

PHOENIX 的架构设计遵循五项核心原则，每一项原则都直接回应了现代密码学面临的关键挑战。这些原则并非抽象的理论概念，而是在每一个实现决策中都得到贯彻的实践准则，从参数选择到代码结构都体现了这些设计哲学。

### 1. 消除单点故障

如果任何一个密码算法或 KDF 被攻破，其他层继续保护数据。这一原则贯穿整个架构设计：三层加密使用三种不同的密码学原语，密钥通过两条独立的 KDF 路径派生，认证通过三个独立的 MAC 机制实现。系统设计时假设任何单一组件都可能在未来被攻破，因此每个安全属性都由至少两个独立组件提供保证。例如，数据保密性由三层加密共同保证，数据完整性由三重认证共同保证，即使其中任意一个或两个组件失败，剩余的组件仍然能够提供有效的保护。

### 2. 量子抗性

KEM 基于 Module-LWE 问题，该问题被认为对量子计算机是困难的。所有对称密钥均为 256 位（通过 Grover 算法提供 128 位量子安全性）。这意味着 PHOENIX 不仅防御当前的经典计算威胁，还防御未来量子计算机可能带来的威胁，包括 Shor 算法（对 RSA/ECC 有效但对格问题无效）和 Grover 算法（将对称密钥安全性减半但仍然不足以构成实际威胁）。在"先收集，后解密"的威胁模型下，这种量子抗性尤为重要——攻击者可能今天收集密文，等待量子计算机成熟后再尝试解密。

### 3. 真正的密码学

每个原语都是真实实现的。KEM 在 Z_q[X]/(X^256+1) 上执行真正的多项式算术，Argon2id 使用真正的 hashlib.blake2b。这些不是占位符或模拟实现——它们实现了与标准参考实现相同的数学框架。虽然纯 Python 实现在性能上不如 C/C++ 实现，但它提供了无可比拟的透明度和可审计性，每一个运算步骤都可以被独立验证。对于生产环境的高吞吐量需求，库支持安装可选的 C 扩展后端来加速关键操作。

### 4. CCA2 安全

Fujisaki-Okamoto 变换确保 KEM 在自适应选择密文攻击下的安全性。这是现代密码学的最高安全标准之一，它保证攻击者即使在能够选择密文并观察解密结果的情况下，也无法获取关于明文的任何有用信息。隐式拒绝机制确保无效密文产生伪随机共享密钥而非错误信号，防止攻击者通过观察错误响应来推断密钥信息。CCA2 安全性是通过将 IND-CPA 安全的公钥加密方案与 Fujisaki-Okamoto 变换相结合来实现的，这一转换的安全性已在随机预言机模型下得到了严格证明。

### 5. 三重认证

三个独立的 MAC 必须全部验证通过才能成功解密。这种设计确保了即使攻击者成功伪造了其中一个认证标签，另外两个标签仍然会阻止解密操作。库刻意不透露哪个标签验证失败，防止攻击者针对特定层进行定向攻击。三重认证机制提供的总认证强度为 128+128+256 = 512 位，远超任何实际需求，但更重要的是它提供了防御多样性——三种认证机制基于完全不同的密码学假设，使得同时攻破所有三种机制的概率极低。

## 密码学原语

### 1. Module-LWE KEM（密钥封装）

Module-LWE KEM 是 PHOENIX 架构的基石，它提供了量子安全的密钥交换能力。该实现基于与 NIST 后量子密码标准 ML-KEM (Kyber-768) 相同的数学框架，在多项式环上执行真正的算术运算。

**问题**：环 Z_q[X]/(X^256+1) 上的 Module Learning With Errors

**参数**（Kyber-768 等效）：
- 环：Z_3329[X]/(X^256+1)
- 模秩：k = 3
- 噪声：CBD(eta1=2, eta2=2)
- 压缩：du=10, dv=4

这些参数经过 NIST 后量子密码标准化过程的严格评估，在安全性和效率之间取得了最佳平衡。素数 q=3329 的选择尤为精妙——它是最小的满足 q ≡ 1 (mod 512) 的素数，这一性质使得数论变换（NTT）可以在该环上高效执行。模秩 k=3 对应 NIST Level 3 安全等级，提供约 192 位经典安全性和 128 位量子安全性。中心二项分布参数 eta1=eta2=2 是保守的选择，确保解密正确率极高同时保持足够的安全余量。

**密钥大小**：
- 公钥：~1184 字节
- 私钥：~2400 字节
- 密文：~1088 字节
- 共享密钥：32 字节

**安全性**：~192 位经典，~128 位量子（NIST Level 3）

**操作**：
- `KeyGen() → (pk, sk)`：使用多项式算术生成密钥对
- `Encaps(pk) → (ct, K)`：封装 32 字节共享密钥
- `Decaps(ct, sk) → K`：使用 FO 变换解封装

### 2. ChaCha20-Poly1305 AEAD

ChaCha20-Poly1305 是第1层加密的核心原语，提供认证加密功能。ChaCha20 由 Daniel Bernstein 设计，是一种不依赖表查找的流密码，因此天然免疫缓存时序攻击。

**算法**：ChaCha20 流密码 + Poly1305 MAC

**属性**：
- 256 位密钥（128 位量子安全性）
- 96 位 nonce
- 128 位认证标签
- 恒定时间执行（无表查找）
- 20 轮四分之一轮运算

**为何选择 ChaCha20？** ChaCha20 免疫可能影响软件中 AES 的缓存时序攻击。它提供了与 AES 不同的数学结构，确保 AES 分析方面的突破不会影响此层。ChaCha20 的四分之一轮运算基于模加法、按位异或和循环左移三种基本操作，这些操作在所有平台上都具有恒定时间执行特性。20 轮的设计远超已知的最佳攻击所能穿透的轮数（目前为 7 轮），提供了极大的安全余量。Poly1305 作为一次性 MAC，在密钥唯一使用时提供信息论级别的认证保证。

### 3. AES-256-GCM AEAD

AES-256-GCM 是第2层加密的核心原语，利用了 AES 作为史上分析最广泛的分组密码的地位。GCM 模式通过将计数器模式加密与 GHASH 认证结合，提供了高效的认证加密方案。

**算法**：高级加密标准（256位）伽罗瓦/计数器模式

**属性**：
- 256 位密钥（128 位量子安全性）
- 96 位 IV
- 128 位认证标签
- 现代 CPU 上的硬件加速（AES-NI）
- 14 轮 SubBytes/ShiftRows/MixColumns/AddRoundKey

**为何选择 AES？** AES 是历史上分析最为广泛的分组密码。硬件加速使其成为现代平台上最快的选项。提供了与 ChaCha20 不同的攻击面。AES-256 的 14 轮设计提供了巨大的安全余量——目前已知的最佳攻击只能穿透不到一半的轮数。GCM 模式的 GHASH 函数在伽罗瓦域 GF(2^128) 上运算，可以高效地并行处理多个数据块，在现代处理器上实现极高的吞吐量。

### 4. SHA3-256 密钥流

SHA3-256 密钥流是第3层加密的核心原语，充当"量子安全网"。SHA3 基于 Keccak 海绵构造，没有量子算法可利用的代数结构。

**算法**：基于 SHA3-256 哈希的流密码

**构造**：keystream[i] = SHA3-256(seed || counter || nonce)

**属性**：
- 256 位种子
- 256 位原像抵抗力（经典）
- 128 位原像抵抗力（量子，通过 Grover 算法）
- 基于海绵构造（Keccak）

**为何选择 SHA3？** 作为哈希函数，SHA3 没有量子算法可利用的代数结构（与 AES 在 GF(2^8) 上的代数结构不同）。该层充当"量子安全网"。SHA3 的海绵构造在吸收阶段和挤压阶段之间提供了清晰的分离，任何对输出的观察都不会泄露关于内部状态的信息。Keccak 在 SHA-3 竞赛中击败了所有其他候选方案，经过了密码学界长达数年的深入分析，其安全性得到了广泛认可。

### 5. Argon2id 密码哈希

Argon2id 是 PHOENIX 中密码哈希的核心原语，符合 RFC 9106 标准，使用 Python 标准库中的 hashlib.blake2b 作为哈希后端。

**算法**：Argon2id（RFC 9106）配合 hashlib.blake2b

**默认参数**：
- 时间开销：4 次迭代
- 内存开销：102,400 KB（100 MB）
- 并行度：2 条通道
- 哈希长度：32 字节
- 盐值长度：16 字节

**为何选择 Argon2id？** 推荐的密码哈希算法。混合模式提供侧信道抵抗（Argon2i 的数据无关寻址在第一遍中使用）和 GPU 抵抗（Argon2d 的数据依赖寻址在后续遍中使用）。100 MB 的内存开销使得 GPU 和 ASIC 攻击者的并行效率大幅降低——一块拥有 8 GB 内存的 GPU 最多只能同时处理约 80 个哈希计算，与可以同时处理数百万次 SHA256 计算相比，攻击成本提高了数个数量级。

## PHOENIX 协议

### 加密流程

PHOENIX 加密流程由五个顺序阶段组成，每个阶段都建立在前一阶段的输出之上，形成了一条从明文到密文的完整安全链条。这种顺序处理确保了每个阶段都为数据增加了独立的安全层，使得最终密文的安全性远超任何单阶段加密方案。

```
输入: plaintext, public_key, associated_data

1. 填充
   ├── 生成随机填充长度（16-256 字节）
   └── padded = LE32(len(plaintext)) || plaintext || random_padding

2. KEM 封装
   ├── 生成随机消息 μ（32 字节）
   ├── 派生 r = SHAKE256(μ || H(pk))
   ├── 计算 u = A^T·r + e1（多项式算术）
   ├── 计算 v = t^T·r + e2 + Encode(μ)
   ├── ciphertext_kem = Compress(u, du) || Compress(v, dv)
   └── shared_secret = SHA3-256(K_bar || H(ciphertext_kem))

3. 密钥派生
   ├── salt = Random(32 字节)
   ├── 路径 1: HKDF-SHA3-256(ss, salt, "ChaCha20") → key_chacha, nonce_chacha
   ├── 路径 2: HKDF-BLAKE2b(ss, salt, "AES256") → key_aes, iv_aes
   └── 路径 3: HKDF-SHA3-256(ss, salt, "HMAC") → key_hmac, seed_sha3

4. 三层级联加密
   ├── 第1层: ChaCha20-Poly1305(key_chacha, padded, nonce_chacha, aad)
   │   → ct_chacha, tag_poly1305
   ├── 第2层: AES-256-GCM(key_aes, ct_chacha||tag_poly1305, iv_aes, aad)
   │   → ct_aes, tag_gcm
   ├── 第3层: SHA3-Keystream(seed_sha3, len(ct_aes||tag_gcm))
   │   → ct_sha3 = XOR(ct_aes||tag_gcm, keystream)
   └── 外层认证: HMAC-SHA3-256(key_hmac, salt||nonce||ct_sha3||aad)
       → tag_hmac

5. 消息构造
   └── output = "QHC2" || version || len(kem_ct) || salt || kem_ct || cascade_ct
```

### 解密流程

解密流程是加密流程的精确逆过程，但增加了额外的安全检查。最重要的是，解密时首先验证最外层的 HMAC-SHA3-256 标签，然后再逐层向内验证和解密。这种"先验证后解密"的模式防止了填充预言攻击，因为攻击者无法通过观察解密错误来获取关于内部明文的信息。如果任何认证检查失败，库会立即拒绝整个解密操作，并且不透露具体是哪一层失败，防止攻击者针对特定层进行定向分析。

```
输入: encrypted_message, private_key, associated_data

1. 解析消息
   └── 提取: magic, version, salt, kem_ciphertext, cascade_ciphertext

2. KEM 解封装（配合 FO 变换）
   ├── 解析私钥组件 (s, pk, H(pk), z)
   ├── 从密文解压缩 u, v
   ├── 计算 μ' = Decode(v - s^T·u)
   ├── 使用 μ' 重新加密以验证（FO 检查）
   ├── 若匹配: shared_secret = SHA3-256(K_bar || H(ct))
   └── 若不匹配: shared_secret = SHA3-256(z || H(ct)) [隐式拒绝]

3. 密钥派生（与加密相同）

4. 三层级联解密
   ├── 验证 HMAC-SHA3-256 标签（第3层认证）
   ├── 撤销 SHA3-Keystream XOR
   ├── 验证 AES-256-GCM 标签（第2层认证）
   ├── 解密 AES-256-GCM
   ├── 验证 Poly1305 标签（第1层认证）
   ├── 解密 ChaCha20
   └── 移除填充

5. 输出
   └── plaintext
```

## 密钥封装（KEM）

### 多项式算术

所有运算在多项式环 R_q = Z_q[X]/(X^256+1) 上进行，其中 q=3329。该环的结构是 ML-KEM (Kyber) 安全性的数学基础——多项式乘法在这个环上的困难性是 Module-LWE 假设的核心。

**关键操作**：
- 加法：系数对模 q 相加
- 乘法：教科书乘法配合 X^256+1 约化
- 当次数 >= 256 时，通过 X^256 = -1 进行约化（负循环）

**示例**：
```
(a * b)[k] = sum(a[i]*b[j]) 对所有 i+j=k (mod 256)，当 i+j >= 256 时符号翻转
```

这种负循环约化是多项式环 R_q 的核心特征。当两个多项式相乘时，乘积的次数可能超过 255，此时需要利用关系式 X^256 = -1 将高次项折减到低次项。具体来说，X^256 被替换为 -1，X^257 被替换为 -X，依此类推。这种折减方式确保了乘法运算始终在环内封闭，同时保持了代数结构的一致性。

### 中心二项分布（CBD）

噪声从 CBD(eta) 中采样，其中 eta=2。CBD 是 ML-KEM 中引入噪声的标准方法，其输出的分布范围有限且以零为中心，这保证了加密和解密之间的微小误差不会导致解密失败。

```
对每个系数:
  1. 读取 2*eta 个随机比特
  2. 拆分为 a_bits（eta 比特）和 b_bits（eta 比特）
  3. coefficient = sum(a_bits) - sum(b_bits)
  → 结果: {-2, -1, 0, 1, 2}（当 eta=2 时）
```

CBD 的选择而非连续高斯分布是有实际原因的：CBD 可以从均匀随机比特高效采样，不需要拒绝采样或浮点运算，这使得实现更加简单且恒定时间。当 eta=2 时，每个系数的取值范围仅为 {-2, -1, 0, 1, 2}，这限制了噪声的幅度，确保解密时的误差不会超过纠错编码的容忍范围。同时，eta=2 的参数选择经过 NIST 的严格安全分析，确认在可预见的经典和量子计算能力下仍能提供足够的安全余量。

### Fujisaki-Okamoto 变换

Fujisaki-Okamoto (FO) 变换是 PHOENIX KEM 安全性的关键保障，它将一个 IND-CPA 安全的公钥加密方案转换为 IND-CCA2 安全的密钥封装机制。这一变换在随机预言机模型下的安全性已被严格证明，是当前后量子密码标准化中广泛采用的技术。

确保 CCA2 安全性，方法是使 KEM 的加密在给定消息时是确定性的：

```
封装:
  μ ← Random(32)
  (r, K_bar) ← G(μ || H(pk))  [从 μ 确定性派生]
  ct ← Enc(pk, μ; r)  [使用从 μ 派生的随机性加密]
  K ← KDF(K_bar || H(ct))

解封装:
  μ' ← Dec(sk, ct)
  重新加密: ct' ← Enc(pk, μ'; G(μ' || H(pk)))
  若 ct == ct': K = KDF(K_bar || H(ct))
  否则: K = KDF(z || H(ct))  [隐式拒绝 - 无错误信号]
```

隐式拒绝确保攻击者无法得知密文是否有效，防止选择密文攻击。当解封装检测到密文不合法时，它不会返回错误信号，而是使用一个预先存储的随机值 z 和密文哈希来派生一个伪随机共享密钥。攻击者无法区分这个伪随机密钥和真实密钥，因此无法通过提交精心构造的密文并观察响应来获取任何关于密钥的信息。这种隐式拒绝机制是 FO 变换安全性的核心组成部分，没有它，攻击者可以通过提交修改过的密文并观察是否产生错误来逐步恢复密钥信息。

## 三层级联加密

### 为何需要三层？

三层级联加密的设计基于一个核心安全原则：纵深防御。在密码学的历史中，曾经被认为安全的算法后来被攻破的案例屡见不鲜——从 DES 的密钥长度不足到 MD5 的碰撞攻击，再到近年来对某些分组密码模式的实际攻击。三层级联确保了即使未来某个密码算法被攻破，数据仍然受到其余层的保护。

| 攻击场景 | 第1层 (ChaCha20) | 第2层 (AES) | 第3层 (SHA3) | 数据安全？ |
|---------|-----------------|------------|-------------|----------|
| AES 被攻破 | 仍然受保护 | 已被攻破 | 仍然受保护 | 是 |
| ChaCha20 被攻破 | 已被攻破 | 仍然受保护 | 仍然受保护 | 是 |
| 两种对称密码均被攻破 | 已被攻破 | 已被攻破 | 仍然受保护 | 是 |
| 量子计算机 | 128位安全 | 128位安全 | 128位安全 | 是 |
| 缓存时序 | 免疫 | 潜在脆弱 | 免疫 | 是 |

### 密钥独立性

每一层的密钥来自**独立的 KDF 路径**：

```
shared_secret ──┬── HKDF-SHA3-256 ──→ ChaCha20 密钥 + nonce
                ├── HKDF-BLAKE2b ──→ AES-256 密钥 + IV
                └── HKDF-SHA3-256 ──→ HMAC 密钥 + SHA3 种子
```

即使 SHA3 被攻破，BLAKE2b 路径仍然保护 AES 层。即使 SHA3 和 BLAKE2b 都被攻破，密钥仍然源自量子安全的 KEM 共享密钥。密钥独立性的保证是通过两个机制实现的：第一，不同的哈希函数（SHA3 和 BLAKE2b）提供了数学层面的分离，因为攻破一个哈希函数不会对另一个产生任何影响；第二，相同的哈希函数配合不同的域分离信息字符串（info string）提供了密码学层面的分离，确保从一条路径的输出无法推导出另一条路径的输出。

此外，密钥的独立性还保证了层与层之间不存在相关密钥攻击的可能。在相关密钥攻击中，攻击者利用不同密钥之间的数学关系来获取优势。由于 PHOENIX 的三层密钥通过完全独立的 KDF 路径派生，即使攻击者获取了某一层的密钥，也无法利用这一信息来简化对其余层的攻击。

## 密钥派生管线

### HKDF-SHA3-256

标准 HKDF（RFC 5869）使用 SHA3-256 作为哈希函数。HKDF 是一个两阶段的密钥派生框架：提取阶段将输入密钥材料与盐值混合以产生伪随机密钥，扩展阶段将伪随机密钥扩展为所需长度的输出密钥材料。这种两阶段设计确保了即使输入密钥材料的熵分布不均匀，最终的输出仍然具有均匀的伪随机分布。

- 提取：PRK = HMAC-SHA3-256(salt, IKM)
- 扩展：OKM = T(1) || T(2) || ... 其中 T(i) = HMAC-SHA3-256(PRK, T(i-1) || info || i)

### HKDF-BLAKE2b

非标准但安全的 KDF，使用 BLAKE2b 作为核心哈希函数。BLAKE2b 是 BLAKE 哈希函数的优化版本，在速度上超过 MD5 和 SHA-1，同时提供了与 SHA-3 相当的安全级别。使用 BLAKE2b 作为替代 KDF 的目的是提供与 SHA3 路径完全独立的密钥派生路径。

- 提取：PRK = BLAKE2b(IKM, key=salt)
- 扩展：T(i) = BLAKE2b(T(i-1) || info || i, key=PRK)

这提供了**独立的** KDF 路径。即使 SHA3 存在致命缺陷，BLAKE2b 仍然提供安全的密钥派生。BLAKE2b 基于 ChaCha 流密码的简化版本，其内部结构与 SHA3 的海绵构造完全不同，因此不存在同时影响两者的密码分析攻击。这种多样性是纵深防御策略在密钥派生层面的体现——正如三层加密使用三种不同的密码算法一样，密钥派生也使用两种不同的哈希函数。

## 密码哈希

### Argon2id 构造

Argon2id 是 RFC 9106 推荐的密码哈希算法，它是 Argon2i 和 Argon2d 的混合体。在第一遍遍历中使用数据无关寻址（Argon2i 模式），在后续遍历中使用数据依赖寻址（Argon2d 模式）。这种混合策略同时提供了侧信道抵抗和 GPU 抵抗能力。

```
H0 = BLAKE2b(LE32(p) || LE32(T) || LE32(m) || LE32(t) || LE32(v) || LE32(y) ||
              LE32(len(pwd)) || pwd || LE32(len(salt)) || salt ||
              LE32(0) || LE32(0))

内存填充（2 遍 - 混合模式）:
  第0遍, 切片0: 数据无关寻址（Argon2i 模式）
  其他所有: 数据依赖寻址（Argon2d 模式）

最终化:
  Final block = 所有通道最后块的 XOR
  Output = BLAKE2b-long(Final block, hash_length)
```

**关键修复**：使用 Python 的 `hashlib.blake2b`（真正的 C 实现）替代了损坏的纯 Python 实现。这确保了与 Argon2 RFC 9106 规范一致的哈希计算。在 v1 版本中，纯 Python 的 Blake2b 实现存在多个正确性缺陷，导致哈希输出与参考实现不匹配。v2.0 通过直接使用 Python 标准库中经过审计的 C 实现彻底解决了这一问题，不仅修复了正确性问题，还显著提升了性能。

## 消息格式

Q-HybridCrypt v2.0 的消息格式采用层次化的嵌套结构，外层包含协议元数据和 KEM 密文，内层包含级联加密的结果。这种设计使得消息解析可以分阶段进行——先验证协议头和版本号，再提取 KEM 密文进行解封装，最后处理级联加密的密文。固定的协议头 "QHC2" 和版本号 0x0002 允许接收方快速识别消息格式，而 KEM 密文长度字段则支持变长密文，为未来的参数升级预留了灵活性。

```
Q-HybridCrypt v2 消息:
┌──────────────┬────────────┬────────────────┬────────────────┬─────────────────┐
│ Magic (4B)   │ Version(2B)│ KEM CT Len(2B) │ Salt (32B)     │ KEM CT (~1088B) │
│ "QHC2"       │ 0x0002     │ LE16           │ Random         │ Lattice CT      │
├──────────────┴────────────┴────────────────┴────────────────┼─────────────────┤
│ 级联密文                                                     │                 │
│ ┌─────────────────────────────────────────────────────────┐ │                 │
│ │ SHA3-Keystream 加密数据（变长）                          │ │                 │
│ │ ┌───────────────────────────────────────────────────┐  │ │                 │
│ │ │ AES-IV (12B) || AES-CT || AES-Tag (16B)          │  │ │                 │
│ │ │ ┌─────────────────────────────────────────────┐  │  │ │                 │
│ │ │ │ ChaCha20-CT || Poly1305-Tag (16B)           │  │  │ │                 │
│ │ │ │ ┌─────────────────────────────────────────┐ │  │  │ │                 │
│ │ │ │ │ LE32(orig_len) || plaintext || padding  │ │  │  │ │                 │
│ │ │ │ └─────────────────────────────────────────┘ │  │  │ │                 │
│ │ │ └─────────────────────────────────────────────┘  │  │ │                 │
│ │ └───────────────────────────────────────────────────┘  │ │                 │
│ └─────────────────────────────────────────────────────────┘ │                 │
│ HMAC-SHA3-256 Tag (32B)                                     │                 │
└─────────────────────────────────────────────────────────────┴─────────────────┘
```

消息格式的嵌套结构直接反映了三层级联加密的处理顺序。最内层是填充后的明文，被 ChaCha20-Poly1305 加密后产生密文和 Poly1305 标签；中间层是 ChaCha20 的输出，被 AES-256-GCM 加密后产生密文和 GCM 标签；最外层是 AES-GCM 的输出，被 SHA3-Keystream XOR 加密，并附上 HMAC-SHA3-256 标签。这种从内到外的嵌套结构确保了解密时必须从外到内逐层剥开，先验证最外层认证再解密内层数据，有效防止了各类预言攻击。

## 参数选择

### KEM 参数（Kyber-768 级别）

KEM 参数的选择经过了 NIST 后量子密码标准化过程中多年的评估和优化。每个参数都有明确的数学或工程理由，在安全性和效率之间取得了经过验证的平衡。

| 参数 | 值 | 理由 |
|------|-----|------|
| n | 256 | 标准多项式次数，平衡安全性和效率 |
| q | 3329 | 满足 q ≡ 1 (mod 512) 的最小素数，支持 NTT |
| k | 3 | NIST Level 3 对应的模秩 |
| eta1 | 2 | 保守的噪声参数，确保解密正确率 |
| eta2 | 2 | 保守的噪声参数 |
| du | 10 | u 分量的压缩位数（平衡大小与正确性） |
| dv | 4 | v 分量的压缩位数 |

### 对称密码参数

对称密码参数的选择遵循最高安全标准。所有密钥均为 256 位，不仅提供了 256 位的经典安全性，还通过 Grover 算法提供了 128 位的量子安全性。IV 和 nonce 的大小遵循各自算法的标准推荐值，确保在每消息独立 KEM 封装的条件下不会出现重用问题。

| 参数 | 值 | 理由 |
|------|-----|------|
| AES 密钥 | 256 位 | 最高 AES 安全性（128位量子） |
| ChaCha20 密钥 | 256 位 | 标准安全性（128位量子） |
| GCM IV | 96 位 | NIST SP 800-38D 推荐 |
| GCM 标签 | 128 位 | 标准认证强度 |
| Poly1305 标签 | 128 位 | 标准 Poly1305 输出 |
| HMAC 标签 | 256 位 | SHA3-256 完整输出 |
| 盐值 | 256 位 | 每消息随机生成 |
| 填充 | 16-256 字节 | 隐藏明文长度 |

### Argon2id 参数

Argon2id 的默认参数遵循 OWASP（开放 Web 应用安全项目）的推荐标准，在安全性和可用性之间取得了实用的平衡。100 MB 的内存开销对现代服务器来说是可以接受的，同时对 GPU 和 ASIC 攻击者构成了显著的并行化障碍。

| 参数 | 值 | 理由 |
|------|-----|------|
| 时间开销 | 4 | OWASP 推荐 |
| 内存开销 | 102,400 KB | 100 MB，平衡安全性和可用性 |
| 并行度 | 2 | 多核利用 |
| 哈希长度 | 32 字节 | 256 位输出 |
| 盐值长度 | 16 字节 | RFC 9106 推荐 |

## 合规性

Q-HybridCrypt v2.0 的设计遵循以下标准和规范的原则，确保其在密码学工程实践中的正确性和可靠性。需要注意的是，Q-HybridCrypt 实现了与 ML-KEM (Kyber) 相同的数学框架，但并非经过 NIST 认证的实现。对于需要 NIST 认证的应用场景，请使用经过验证的 ML-KEM 实现。

- NIST 后量子密码标准化原则
- FIPS 197 (AES)
- NIST SP 800-38D (GCM)
- RFC 8439 (ChaCha20-Poly1305)
- RFC 9106 (Argon2)
- RFC 5869 (HKDF)
