# Q-HybridCrypt v2.0 - 安全性分析

## 威胁模型

### 对手能力

Q-HybridCrypt v2.0 "PHOENIX" 的安全设计基于一个全面的威胁模型，涵盖了经典计算对手和量子计算对手可能具备的各种能力。威胁模型定义了系统设计为抵抗的攻击类型以及每种攻击的抵抗机制。理解威胁模型的范围和边界对于正确评估 PHOENIX 在特定应用场景中的适用性至关重要——没有任何密码学系统可以抵抗所有可能的攻击，明确威胁边界是做出合理安全决策的基础。

| 能力 | 经典对手 | 量子对手 |
|------|---------|---------|
| 暴力破解 | 2^192（不可行） | 2^128（不可行） |
| Shor 算法 | 不适用 | 仅对 RSA/ECC 有效 |
| Grover 算法 | 不适用 | 256 位密钥对应 2^128 |
| 选择明文 | AEAD 处理 | AEAD 处理 |
| 选择密文 | FO 变换处理 | FO 变换处理 |
| 侧信道（时序） | 缓解（ChaCha20, 恒定时间） | 相同 |
| 缓存攻击 | 缓解（ChaCha20） | 相同 |
| GPU/ASIC 暴力破解 | 缓解（Argon2id） | 相同 |

经典对手的计算能力受限于物理定律——即使使用全球所有的计算资源，2^192 次运算在可预见的未来也是不可能完成的。量子对手虽然可以通过 Shor 算法在多项式时间内解决整数分解和离散对数问题，但这些算法对基于格问题（Module-LWE）的密码系统无效。Grover 算法可以将对称密钥搜索的复杂度从 2^n 降低到 2^(n/2)，但 PHOENIX 使用的 256 位对称密钥即使在 Grover 攻击下仍然提供 128 位的安全性，远远超出实际攻击能力。

### 对手类型详述

**被动窃听者**：只能观察密文和公开信息（公钥、消息格式），无法修改或注入数据。PHOENIX 的三层级联加密确保被动窃听者无法从密文中获取关于明文的任何信息。即使窃听者拥有无限存储空间来记录所有密文，密文与明文之间的统计独立性保证了信息的保密性。

**主动攻击者**：可以修改、重放或注入密文，并观察解密结果。PHOENIX 的三重认证机制（Poly1305 + GCM + HMAC-SHA3）确保任何对密文的修改都会被检测到，Fujisaki-Okamoto 变换则防止了选择密文攻击。主动攻击者无法通过提交精心构造的密文来获取关于密钥或明文的有用信息，因为任何无效密文都会触发隐式拒绝机制。

**量子攻击者**：拥有大规模容错量子计算机，可以运行 Shor 算法和 Grover 算法。PHOENIX 的 Module-LWE KEM 对 Shor 算法免疫，256 位对称密钥通过 Grover 算法提供 128 位安全性。量子攻击者无法利用量子并行性来加速对 Module-LWE 问题的求解，因为该问题不具有量子算法可利用的特定代数结构。

## 安全性证明

以下提供 PHOENIX 核心安全性质的直觉性证明概要。这些证明虽然不是形式化的数学证明，但基于密码学界广泛接受的安全归约方法，将 PHOENIX 的安全性归约到被广泛研究的底层困难问题上。每个证明都遵循相同的范式：假设底层问题是困难的，那么攻破 PHOENIX 的安全性在计算上等价于解决该困难问题。

### 1. KEM 安全性（IND-CCA2）

KEM 通过 Fujisaki-Okamoto 变换在 Module-LWE 假设下实现 CCA2 安全。

**定理**：如果底层 PKE 方案在 Module-LWE 下是 IND-CPA 安全的，则经 FO 变换的 KEM 是 IND-CCA2 安全的。

**证明概要**：
- Module-LWE 被认为对经典和量子对手都是困难的。该问题的困难性基于随机自归约性——即使对手获得了多个实例的解答，新的实例仍然与随机实例不可区分。目前已知最佳的经典和量子攻击算法都需要指数级时间，具体来说，对于 Kyber-768 级别的参数，最佳攻击约需 2^192 次经典运算或 2^128 次量子运算。
- FO 变换使加密在给定消息 μ 时是确定性的。这种确定性是 CCA2 安全的关键——因为加密的随机性是从消息本身派生的，攻击者无法通过选择不同的随机性来创建相关的密文。
- 任何修改过的密文将无法通过重加密验证。当解封装器解密出消息 μ' 后，它会使用 μ' 重新加密并检查结果是否与原始密文匹配。如果不匹配，说明密文被篡改。
- 隐式拒绝防止通过错误响应泄露信息。攻击者无法区分"密文无效导致的伪随机输出"和"正常解封装产生的真实输出"，因此无法从解密预言机的响应中获取任何关于密钥的信息。
- 共享密钥同时依赖于消息 μ 和密文哈希。这种双重依赖确保了共享密钥的唯一性和不可预测性。

### 2. 级联安全性

**定理**：三层级联加密的安全性至少与最强的剩余层相当。

**证明概要**：
- 每层使用独立派生的密钥。三个密钥通过不同的 KDF 路径从共享密钥派生，密钥之间的计算独立性由哈希函数的伪随机性保证。
- 密钥通过独立的 KDF 路径（SHA3-256 和 BLAKE2b）派生。不同哈希函数的使用提供了数学层面的分离——攻破一个哈希函数对另一个没有影响。
- 攻破一个密钥不帮助恢复其他密钥。由于密钥的独立性，已知一个密钥不会减少搜索其他密钥的空间。
- 要恢复明文，对手必须攻破全部三层。这是级联加密的核心安全保证——数据的安全性由最弱的未被攻破的层决定。
- 整体安全性为：max(security_layer1, security_layer2, security_layer3)。这意味着即使两层被攻破，数据仍然受到第三层的保护。

级联安全性的直觉可以用一个简单的类比来理解：想象三个独立的保险柜嵌套放置，每个保险柜使用不同的锁和不同的钥匙。要取出最内层的物品，攻击者必须依次打开所有三个保险柜，而打开外层保险柜并不能简化打开内层保险柜的难度，因为每个锁的机制完全独立。

### 3. 密钥分离

**定理**：从不同 KDF 路径派生的密钥在计算上是独立的。

**论证**：
- 路径 1 使用 HKDF-SHA3-256，域分离信息为 "Q-HybridCrypt-v2-ChaCha20-Key"
- 路径 2 使用 HKDF-BLAKE2b，域分离信息为 "Q-HybridCrypt-v2-AES256-Key"
- 路径 3 使用 HKDF-SHA3-256，域分离信息为 "Q-HybridCrypt-v2-HMAC-Key"
- 不同的哈希函数（SHA3 与 BLAKE2b）提供数学分离。SHA3 基于 Keccak 海绵构造，BLAKE2b 基于 ChaCha 流密码的简化版本，两者内部结构完全不同，不存在同时影响两者的密码分析攻击。
- 相同哈希函数配合不同信息字符串提供域分离。HKDF 的扩展阶段将 info 字符串作为输入之一，不同的 info 字符串确保了即使使用相同的伪随机密钥，输出的密钥材料也是完全不同的。
- 从一个密钥恢复另一个密钥将需要攻破底层哈希函数。这是密码学中标准的归约论证——如果攻击者能够从一条 KDF 路径的输出推导出另一条路径的输出，那么攻击者就可以区分哈希函数的输出与随机值，这与哈希函数的伪随机性假设矛盾。

## 抗攻击能力

### 经典密码分析

PHOENIX 的三层级联架构对经典密码分析方法提供了多层防御。每一层都针对不同类型的密码分析攻击具有内在的抵抗力，三层组合后的安全性远超任何单层方案。以下表格详细列出了各类经典攻击的抵抗机制和原理。

| 攻击 | 抵抗能力 | 机制 |
|------|---------|------|
| 线性密码分析 | 抵抗 | ChaCha20（无线性逼近） |
| 差分密码分析 | 抵抗 | AES-256（14轮，完全扩散） |
| 相关密钥攻击 | 抵抗 | 每层独立密钥 |
| 中间人攻击 | 抵抗 | 三层需要全部攻破 |
| 生日攻击 | 抵抗 | 256 位哈希输出 |

**线性密码分析**通过寻找密码算法的线性近似来恢复密钥信息。ChaCha20 的四分之一轮运算基于模加法、按位异或和循环左移，这些操作的非线性度极高，不存在可利用的线性逼近。即使 AES 在理论上存在线性逼近（尽管 14 轮 AES 的线性偏差极小），ChaCha20 层的存在也完全阻断了线性密码分析的攻击路径。

**差分密码分析**通过追踪明文差分在加密过程中的传播来恢复密钥。AES-256 的 14 轮设计提供了完全扩散——输入中任何一位的改变在两轮之后就会影响输出中的每一位——使得高概率差分路径无法存在超过几轮。ChaCha20 和 SHA3 层的额外加密进一步消除了任何残余的差分传播可能性。

**相关密钥攻击**利用不同密钥之间的数学关系来获取优势。PHOENIX 的三层密钥通过独立的 KDF 路径派生，即使攻击者获取了某一层的密钥，也无法利用这一信息来简化对其余层的攻击，因为密钥之间不存在可利用的相关性。

### 量子密码分析

量子密码分析代表了密码学面临的全新挑战。与传统计算不同，量子计算利用量子力学的叠加和纠缠特性来执行某些计算任务，可以在多项式时间内解决某些经典计算需要指数时间的问题。PHOENIX 的设计充分考虑了量子攻击者的能力，确保在量子计算威胁下仍然提供足够的安全保证。

| 攻击 | 抵抗能力 | 机制 |
|------|---------|------|
| Shor 算法 | 抵抗 | Module-LWE（非因式分解/离散对数） |
| Grover 算法 | 128 位安全性 | 256 位对称密钥 |
| 量子周期查找 | 抵抗 | LWE 无可利用的周期性 |
| 量子随机游走 | 未知 | 保守参数缓解风险 |

**Shor 算法**是量子计算对传统公钥密码的最大威胁。该算法可以在多项式时间内解决整数分解和离散对数问题，从而完全攻破 RSA 和 ECC。然而，Shor 算法依赖于找到有限阿贝尔群中的周期结构，而 Module-LWE 问题不具备这种结构。目前没有任何已知的量子算法可以在多项式时间内解决 Module-LWE 问题，这也是 NIST 选择基于格的方案作为后量子密码标准的主要原因。

**Grover 算法**可以将无结构搜索问题的复杂度从 O(N) 降低到 O(√N)。对于对称密钥搜索，这意味着 256 位密钥的有效安全性降低到 128 位。虽然 128 位安全性仍然远超实际攻击能力（2^128 次运算即使在量子计算机上也需要不可接受的时间），但这是 PHOENIX 选择 256 位对称密钥而非 128 位的直接原因。

**量子周期查找**是 Shor 算法的核心技术，它利用量子傅里叶变换来发现函数的隐藏周期。Module-LWE 问题中的噪声使得这种周期查找方法无法直接应用，因为噪声破坏了函数的精确周期性。目前没有已知的方法可以在存在噪声的情况下有效地使用量子周期查找来解决 LWE 类问题。

**量子随机游走**是一种更复杂的量子算法框架，其对抗 LWE 的效果目前尚不明确。为应对这种潜在的未知威胁，PHOENIX 采用了保守的参数选择（eta1=2, eta2=2），提供了比最低安全要求更大的安全余量。

### 侧信道攻击

侧信道攻击通过观察密码系统的物理实现特征（如执行时间、功耗、电磁辐射等）来获取密钥信息。这类攻击不攻击密码算法本身的数学结构，而是攻击算法在特定硬件上的实现特征。PHOENIX 对侧信道攻击的防御是部分性的——在软件层面提供了针对时序攻击和缓存攻击的缓解措施，但硬件级别的侧信道攻击超出了纯软件库的防御范围。

| 攻击 | 抵抗能力 | 机制 |
|------|---------|------|
| 时序攻击 | 部分 | ChaCha20 恒定时间; AES 不是 |
| 缓存攻击 | 部分 | ChaCha20 无表查找 |
| 功耗分析 | 未处理 | 需要硬件级对策 |
| 电磁攻击 | 未处理 | 需要硬件级对策 |

**时序攻击**通过测量密码操作的执行时间来推断密钥信息。ChaCha20 的设计天然免疫时序攻击——它的四分之一轮运算仅使用模加法、按位异或和循环左移，这些操作在所有处理器上都具有恒定执行时间。然而，纯 Python 的 AES 实现通过 S-box 表查找执行字节替换，表查找的执行时间可能因缓存命中或未命中而异，这可能泄露密钥信息。幸运的是，ChaCha20 作为级联的第一层，其输出在传递给 AES 之前已经被随机化，这使得针对 AES 的时序攻击更加困难。

**缓存攻击**通过观察处理器的缓存状态变化来推断密钥信息。AES 的 S-box 查找操作使得缓存攻击成为可能——攻击者可以通过观察哪些缓存行被访问来推断密钥的某些位。ChaCha20 不执行任何表查找操作，因此天然免疫缓存攻击。在 PHOENIX 的三层级联中，即使攻击者通过缓存攻击获取了 AES 层的部分密钥信息，仍然需要攻破 ChaCha20 和 SHA3 层才能恢复明文。

### 协议级攻击

协议级攻击针对密码系统的使用方式和协议设计进行攻击，而非密码算法本身的数学结构。这类攻击通常利用协议设计中的逻辑缺陷或实现中的边界条件错误。PHOENIX 对协议级攻击的防御是全面但非穷尽的——某些攻击类型（如重放攻击和密钥冒充）需要在应用层面进行额外的防御。

| 攻击 | 抵抗能力 | 机制 |
|------|---------|------|
| 重放攻击 | 应用责任 | 在 AAD 中使用时间戳/序列号 |
| 选择密文 | 抵抗 | FO 变换 + 三重认证 |
| 填充预言 | 抵抗 | 先验证后解密模式 |
| 密钥冒充 | 未处理 | 使用基于证书的密钥验证 |
| 前向保密 | 部分 | 每消息短暂 KEM |

**重放攻击**通过重新发送之前的有效密文来欺骗接收方。PHOENIX 本身不提供内置的重放检测机制——解密操作是无状态的，无法判断一条密文是否之前已经被解密过。防御重放攻击是应用层的责任，推荐的做法是在关联数据（AAD）中包含时间戳、序列号或消息ID，并在应用层维护一个已处理消息的去重列表。

**选择密文攻击**通过提交精心构造的密文并观察解密行为来获取密钥信息。PHOENIX 的 Fujisaki-Okamoto 变换确保无效密文产生伪随机输出而非错误信号，防止攻击者从解密结果中获取信息。三重认证机制提供了额外的保障——攻击者构造的密文必须同时通过三个独立的认证检查，这在计算上是不可能的。

**填充预言攻击**通过观察填充验证是否成功来逐步恢复明文信息。PHOENIX 采用"先验证后解密"模式——首先验证最外层的 HMAC-SHA3-256 标签，如果验证失败则立即拒绝整条消息，不进入内部层的处理。这种设计确保攻击者无法通过观察内层填充验证的结果来获取信息。

## 抗量子能力

### 量子计算威胁概述

量子计算对传统密码学构成了前所未有的挑战。大型容错量子计算机一旦建成，将能够运行 Shor 算法在多项式时间内分解大整数和计算离散对数，从而完全攻破 RSA、DSA、ECDSA 和 ECDH 等广泛使用的公钥密码系统。对于对称密码，Grover 算法可以将暴力搜索的复杂度从 O(2^n) 降低到 O(2^(n/2))，这意味着 128 位密钥在量子攻击下仅提供 64 位安全性。

然而，量子计算并非万能。Shor 算法只能解决具有特定代数结构的问题（整数分解和离散对数），对于基于格问题、编码问题、多变量方程组问题和哈希问题等后量子密码学基础问题，目前没有已知的高效量子算法。PHOENIX 的 Module-LWE KEM 正是基于这类量子困难问题，因此在量子计算时代仍然保持安全。

### PHOENIX 的量子抗性分析

**KEM 层的量子抗性**：Module-LWE 问题被认为对量子对手是困难的。这一判断基于以下理由：首先，Module-LWE 是一般 LWE 问题的结构化变体，而 LWE 的困难性可以归约到格上的最坏情况困难问题；其次，经过 NIST 后量子密码标准化过程中数年的公开分析，没有发现对 Kyber/ML-KEM 参数的有效量子攻击；最后，PHOENIX 使用的参数（k=3, eta=2）提供了约 128 位的量子安全性，这意味着即使使用理想的量子计算机，攻破 KEM 也需要约 2^128 次量子运算。

**对称层的量子抗性**：所有对称密钥均为 256 位，通过 Grover 算法提供 128 位量子安全性。三层级联进一步增强了量子抗性——即使量子计算机能够加速对某一层的攻击，仍然需要攻破全部三层。128 位的量子安全性意味着即使攻击者拥有一台每秒执行 2^50 次量子运算的量子计算机（远超当前任何技术预期），也需要约 2^78 秒（约 10^16 年）来完成攻击。

**"先收集，后解密"威胁**：当前的量子计算机还无法破解 RSA、ECC 或对称加密，但"先收集，后解密"的威胁是真实存在的。攻击者可能今天收集加密数据，等待量子计算机成熟后再尝试解密。对于具有长保密周期的数据（如医疗记录、金融数据、国家机密），这种威胁尤为紧迫。迁移到 Q-HybridCrypt 确保在 PHOENIX 协议下加密的数据即使面对未来的量子对手也保持安全。

## 已知限制

诚实面对密码学系统的局限性是负责任的安全实践的重要组成部分。以下列出了 Q-HybridCrypt v2.0 "PHOENIX" 的已知限制，以及每项限制的影响范围和可能的缓解措施。了解这些限制有助于您在特定应用场景中做出合理的安全决策。

### 1. 纯 Python 性能

当前实现为纯 Python，使其显著慢于 C/C++ 实现。密钥生成约需 2-5 秒，加密 1 KB 数据约需 1-3 秒。对于高吞吐量应用，这种性能可能是不可接受的。缓解措施包括安装可选的性能后端（`pip install q-hybridcrypt[performance]`）或将密集密码学操作卸载到专用服务。纯 Python 实现的优势在于透明度和可审计性——每一行代码都可以被直接审查，无需依赖外部编译的二进制文件。

### 2. AES 中的时序侧信道

纯 Python AES 实现可能通过表查找（S-box）存在时序变异性。S-box 查找操作在缓存命中和未命中时的执行时间不同，这可能泄露密钥的某些位信息。ChaCha20 层作为第一加密层缓解了此问题——ChaCha20 的输出是伪随机的，使得 AES 的输入不具有攻击者可控制的模式，从而增加了时序攻击的难度。对于需要完全时序安全的应用，建议使用硬件加速的 AES-NI 或安装 `cryptography` 库作为后端。

### 3. 缺乏硬件级侧信道加固

功耗分析和电磁攻击未在本库中处理。这是软件库的固有局限——功耗分析和电磁辐射取决于算法在特定硬件上的物理执行特征，纯软件无法控制这些物理量。对于处理极高价值数据的应用（如金融交易签名、国家机密加密），需要在硬件层面实施侧信道对策，包括恒定时间硬件实现、功耗随机化和电磁屏蔽等。本库作为软件解决方案，其侧信道防御范围仅限于软件可实现的时间恒定化和缓存访问模式优化。

### 4. 内存使用

Argon2id 默认参数使用约 100 MB RAM。这是有意为之的（内存困难性），但可能不适用于受限设备，如嵌入式系统、物联网设备或内存配额有限的容器环境。对于这些场景，可以通过调整 `memory_cost` 参数来降低内存使用量，但请注意这也会降低对 GPU/ASIC 攻击的抵抗能力。例如，将内存开销降低到 10 MB 将使攻击者能够在一块 GPU 上并行处理约 800 个哈希计算，而非仅约 80 个。

### 5. 大密钥尺寸

基于格的 KEM 密钥约为 1-2 KB，显著大于 RSA-2048（256 字节）或 ECC-P256（64 字节）。这是格密码学的固有特征——格问题的安全性依赖于高维空间中的计算困难性，而高维性不可避免地导致密钥和密文的尺寸增大。随着后量子密码学的成熟和优化技术的进步，密钥尺寸有望进一步缩小，但在可预见的未来，格密码学的密钥尺寸仍将显著大于传统公钥密码。

### 6. 无密钥轮换协议

库提供密码学原语但不实现密钥管理或轮换协议。应用必须自行实现密钥的生命周期管理，包括密钥生成、分发、存储、轮换和销毁。密钥轮换是长期运行系统安全性的关键组成部分——定期更换密钥可以限制单次密钥泄露的影响范围。建议应用实现基于时间的自动密钥轮换机制，并维护密钥版本历史以支持旧密文的解密。

## 安全建议

以下安全建议基于 PHOENIX 的威胁模型和已知限制，为实际部署提供具体可操作的指导。每条建议都针对一个特定的安全风险，提供了详细的实施方法和理由。遵循这些建议可以最大限度地发挥 PHOENIX 的安全潜力，同时避免常见的安全配置错误。

### 1. 始终使用 AAD

在关联数据中包含上下文信息（用户ID、时间戳、用途），将密文与其上下文绑定。AAD 经认证但不加密，是防止密文在不同上下文中被误用的关键机制。例如，如果您的系统同时加密用户密码重置令牌和 API 访问令牌，使用不同的 AAD 可以防止攻击者将一种令牌替换为另一种。推荐的 AAD 格式为：`purpose|user_id|timestamp|version`，例如 `b"password_reset|user12345|1700000000|v2"`。

### 2. 绝不重用 nonce

每次加密使用新的 KEM 封装，确保每条消息拥有唯一密钥。不要尝试重用密钥材料。PHOENIX 的设计已经通过每消息独立的 KEM 封装自动确保了 nonce 的唯一性——每次加密操作都会生成新的共享密钥，从中派生的 nonce 也是全新的。违反这一原则（例如，试图在多次加密间共享密钥）将破坏安全性保证，特别是可能导致 ChaCha20 的流密码密钥流重用，这会直接泄露明文的 XOR。

### 3. 使用强密码

Argon2id 保护弱密码但无法补偿极弱的密码。使用长随机密码，建议至少 12 个字符，包含大小写字母、数字和特殊符号。对于程序生成的密码，建议使用至少 16 字节的随机熵。Argon2id 的 100 MB 内存开销将每次密码猜测尝试的成本提高到了 GPU 上约 0.01 美元，但对于像 "password123" 这样的常见密码，攻击者仍然可以通过字典攻击在合理时间内找到答案。密码强度和密码哈希算法的强度是互补的，两者缺一不可。

### 4. 验证公钥

加密前始终验证公钥属于预期的接收方。没有认证的公钥交换容易受到中间人攻击——攻击者可以用自己的公钥替换接收方的公钥，从而解密所有发送给该公钥的消息。密钥替换攻击在没有认证的情况下是可能的。建议使用基于证书的密钥验证或带外密钥交换（如面对面交换公钥指纹）来确保公钥的真实性。

### 5. 安全密钥存储

私钥必须安全存储。生产环境考虑使用硬件安全模块（HSM）进行密钥存储。HSM 提供了防篡改的密钥存储环境，私钥永远不会离开 HSM 的安全边界，所有加密和解密操作都在 HSM 内部完成。如果不使用 HSM，至少应将私钥存储在加密的文件中，并严格限制文件访问权限。绝不要将私钥提交到版本控制系统、存储在日志文件中或通过不安全的通道传输。

### 6. 保持软件更新

关注安全更新和密码学进展，特别是后量子密码学领域。密码学是一个快速发展的领域，新的攻击方法和防御技术不断出现。建议订阅 NIST 的后量子密码学公告、相关密码学会议的论文摘要以及 Q-HybridCrypt 的安全更新通知。在后量子密码学标准化完成后的过渡期内，参数和算法可能需要更新，及时获取这些信息对于维护系统的长期安全性至关重要。

## 与 NIST PQC 标准对比

Q-HybridCrypt v2.0 的 KEM 实现基于与 NIST 后量子密码标准 ML-KEM (Kyber) 相同的数学框架，但在实现级别和认证状态上存在重要差异。下表详细对比了两者的异同，帮助用户理解 Q-HybridCrypt 在后量子密码标准化进程中的定位。

| 方面 | Q-HybridCrypt v2 | ML-KEM (Kyber) 标准 |
|------|------------------|---------------------|
| KEM 基础 | Module-LWE | Module-LWE |
| KEM 安全性 | NIST Level 3 | 可配置 (1/3/5) |
| 对称加密 | 三层级联 | 未指定 |
| CCA2 | FO 变换 | FO 变换 |
| 实现 | 纯 Python | 参考实现为 C |
| 认证状态 | 未经 NIST 认证 | NIST FIPS 203 |

**重要说明**：Q-HybridCrypt 实现了与 ML-KEM (Kyber) 相同的数学框架，但**并非**经过 NIST 认证的实现。Q-HybridCrypt 的 KEM 执行真正的多项式算术和 CBD 噪声采样，与 Kyber-768 的数学运算一致，但在实现细节（如多项式乘法使用教科书算法而非 NTT）和工程保障（如侧信道加固、形式化验证）方面与经过认证的实现存在差距。对于需要 NIST 认证的应用场景，请使用经过验证的 ML-KEM 实现，如 liboqs 或 AWS-LC。

Q-HybridCrypt 的独特价值在于其三层级联加密架构——这是 ML-KEM 标准未涉及的领域。ML-KEM 定义了密钥封装机制，但将对称加密的选择留给了应用。Q-HybridCrypt 不仅提供了量子安全的密钥封装，还在对称加密层面实现了纵深防御，这是任何单一标准实现无法提供的额外安全保证。
